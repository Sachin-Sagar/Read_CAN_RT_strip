Project Debrief: High-Performance CAN Logger

The Debugging Journey: From Bottlenecks to Breakthrough

Our goal was to build a highly reliable logger for both high and low-frequency CAN signals. While we successfully solved major performance challenges, we were left with a persistent bug where 10ms signals were not being logged. This document outlines our troubleshooting process and the final solution.

Stage 1: Eliminating Architectural Bottlenecks

We evolved the application through several stages to achieve the necessary performance:

  • I/O Bottleneck: Moved file writing to a dedicated thread with batch processing to prevent blocking.
  • CPU Bottleneck (GIL): Replaced CPU-intensive decoding threads with a pool of parallel processes to bypass the Global Interpreter Lock.
  • IPC Overhead: Implemented a shared memory array (`multiprocessing.RawArray`) to eliminate slow data serialization between processes, moving from passing large objects to only passing a tiny integer index.
  • Hardware Buffer Overflow: Reduced the `bus.recv()` timeout to a minimum, ensuring the hardware's internal buffer was polled constantly and never overflowed.

After these optimizations, performance was excellent, but the 10ms signals were still missing. This proved the issue was not performance-related.

The Final Breakthrough: The Root Cause

The final problem was a subtle but critical data mismatch. The CAN IDs for our 10ms signals were stored with a leading zero in the signal list (e.g., '0x09a'), but the live CAN message parser generated an unpadded string (e.g., '0x9a').

Because the strings did not match exactly, the dictionary lookup failed, and the high-frequency messages were silently dropped. The low-frequency signals worked because their IDs didn't have this padding discrepancy.

The fix is to standardize on using integer CAN IDs for all lookups, which completely removes any string formatting ambiguity.
